BehaviorTree 核心：节点注册体系与工程本质

框架负责「如何组织任务」，用户实现「任务具体做什么」

一、核心注册规则（关键结论）

节点类型 是否需要用户注册 原因
Action Node ✅ 必须 业务强相关，框架无法预知具体行为
Condition Node ✅ 必须 业务强相关，需自定义条件逻辑
Control Node ❌ 无需 框架内置核心流程控制逻辑（如 Sequence）
Decorator Node ⚠️ 通常无需 基础版内置，仅自定义装饰器需注册

💡 本质区别：
Control/Decorator = 流程控制能力（框架提供）
Action/Condition = 业务能力（用户实现）

二、为什么 Action/Condition 必须注册？

1. XML 中的字符串需要绑定 C++ 实现

框架必须知道：
"FollowPath" → 对应哪个 C++ 类？

2. 注册机制：工厂模式绑定

效果：
XML 中的 <FollowPath/> → 实例化为 FollowPathNode 对象

3. 未注册的后果

框架无法将字符串映射到实际逻辑，行为树无法运行。

三、为什么 Control Node 无需注册？

1. 框架已预置核心控制逻辑

BehaviorTree.CPP 启动时自动注册：

2. 用户直接使用即可

无需额外代码：框架已提供 Sequence 的完整执行规则（如“全成功才成功”）。

四、工程实践：业务逻辑与框架的分离

1. Control Node：通用流程控制（稳定不变）

节点类型 用途 示例场景
Sequence 串行任务（全成功才成功） 导航：检查电量→规划路径→跟踪
ReactiveFallback 动态抢占（高优先级条件触发） 急停：障碍检测→中断导航
PipelineSequence 流水线重规划（Nav2 核心） 边跟踪路径边动态重规划

2. Action/Condition：业务强相关（高度定制）

领域 Action 示例 Condition 示例
机器人导航 ComputePath, FollowPath IsBatteryLow, GoalUpdated
自动驾驶 LaneChange, StopAtTrafficLight IsRedLight, IsObstacleNear
工业机器人 PickObject, MoveArm IsGripperEmpty, IsPartDetected

✨ 关键洞察：
Control Node 是“流程骨架”，Action/Condition 是“业务血肉”
框架提供通用骨架，用户填充领域血肉。

五、Decorator 的特殊性：内置 vs 自定义

类型 注册需求 示例
内置 Decorator ❌ 无需注册 <RetryUntilSuccessful/>, <Timeout/>
自定义 Decorator ✅ 需注册 <MyCustomTimeout/>

六、类比：操作系统级设计哲学

组件 BehaviorTree.CPP 操作系统
核心能力 Control/Decorator 节点 调度器、系统调用接口
用户实现部分 Action/Condition 节点 用户进程（业务逻辑）
注册机制 factory.registerNodeType system call registration
执行单位 Behavior Tree Tick 进程调度周期

本质相同：
框架提供 任务组织能力（如流程控制、生命周期管理），用户实现 领域业务逻辑。

八、终极总结：行为树的工程本质

1. 框架只做三件事

✅ 解析 XML → 构建树结构
✅ 执行 Tick → 按 Control Node 规则调度
✅ 提供基础 Control/Decorator → 实现流程控制语义

2. 用户只需做一件事

💡 注册业务节点 → 填充 Action/Condition 的领域逻辑

3. 最核心原则

“Control Node 定义任务如何执行，Action/Condition 定义任务具体做什么”
行为树的业务能力 100% 来源于用户注册的 Action/Condition 节点
