BehaviorTree 核心：Decorator Node 流程影响

真正的核心不是 Action Node，而是 Control Node 的执行规则。
Decorator Node 虽然：
# 不负责组织多个节点
但它：
# 会深度影响执行流程，改变节点语义。

一、Decorator 最常见的 6 类流程影响

类型 核心影响 示例场景
Retry 失败后重试 路径跟踪失败时自动重试
Repeat 成功后重复执行 巡逻任务循环执行指定次数
Timeout 超时中断执行 等待电梯超时触发失败
ForceSuccess / ForceFailure 强制修改返回值 记录日志但不中断主流程
Inverter 反转节点执行结果 将“检测到障碍”转为“无障碍”逻辑
RateController 限制节点 Tick 频率 导航中控制路径重规划速率

三、Decorator 的本质力量

不修改节点代码，却能彻底改变节点行为：

  1. 返回值控制（如 ForceSuccess）。
  2. 生命周期管理（如 Retry、Repeat）。
  3. 时间约束（如 Timeout）。
  4. 执行频率调节（如 RateController）。
  5. 逻辑反转（如 Inverter）。

五、终极总结

Decorator Node 本质是：
“节点行为变换器”
它通过封装非侵入式修改，赋予节点以下能力：

  1. 定义何时结束执行。
  2. 定制失败/成功的后处理逻辑。
  3. 约束执行时间与频率。
  4. 解耦逻辑与流程控制。

这种设计哲学，正是 BehaviorTree.CPP 在机器人领域实现高度灵活性和可复用性的核心基石。

BehaviorTree.CPP 核心：控制节点语义对比指南

真正的核心不是 Action Node，而是 Control Node 的执行规则

一、Sequence vs Fallback（经典对比）

Sequence（顺序执行）

含义：
一步一步完成任务
逻辑：

A→B→CA \rightarrow B \rightarrow CA→B→C

规则：
一个失败 → 全部失败
适用场景：

  • 导航流程
  • 机器人抓取
  • 工业流水线

Fallback（选择器）

含义：
一个失败就尝试下一个
逻辑：
A OR B OR C
规则：
一个成功 → 立刻停止

BehaviorTree.CPP 核心机制：节点注册体系与工程本质

框架负责「如何组织任务」，用户实现「任务具体做什么」

一、核心注册规则（关键结论）

节点类型 是否需要用户注册 原因
Action Node ✅ 必须 业务强相关，框架无法预知具体行为
Condition Node ✅ 必须 业务强相关，需自定义条件逻辑
Control Node ❌ 无需 框架内置核心流程控制逻辑（如 Sequence）
Decorator Node ⚠️ 通常无需 基础版内置，仅自定义装饰器需注册

💡 本质区别：
Control/Decorator = 流程控制能力（框架提供）
Action/Condition = 业务能力（用户实现）

ESKF的构建流程模板

在实际工程中，我们常常面对“任意组合”的多传感器系统，例如轮速计 + LiDAR、IMU + 编码器、或更复杂的视觉惯性方案。无论传感器如何组合，其核心问题始终是：如何在噪声、偏置与非线性条件下构建一个一致、鲁棒、可递推的状态估计器？ 本博客的目的，就是给出一套可泛化的、可直接落地的 ESKF（误差状态卡尔曼滤波）建模与实现流程。

0. 记号与目标

  • 真值状态：xtrue(t)x^\text{true}(t)xtrue(t)
  • 名义状态：x(t)x(t)x(t)（滤波器内部集成的）
  • 误差状态：δx(t)\delta x(t)δx(t)（线性小量，用来做 KF）

目标：给定连续非线性系统

x˙true=f(xtrue,u,w) \dot{x}^\text{true} = f(x^\text{true}, u, w) x˙true=f(xtrue,u,w)

观测模型

y=h(xtrue,n) y = h(x^\text{true}, n) y=h(xtrue,n)

IMU+轮速计的ESKF简明介绍

（基于右不变误差、IMU 预测 + 轮速计线速度观测）

1. 引言

在移动机器人中，IMU（惯性测量单元）提供高频预测，而轮速计提供低频但稳定的前向速度信息。
二者结合的 ESKF（Error-State Kalman Filter）能够实现快速、稳定且低漂移的状态估计。

本篇主要聚焦：
